## Context

Datacore is a tenant-scoped LanceDB storage library used across NeoApex. It currently has no HTTP interface — consuming projects import it directly. The add-student-entry feature requires admindash (a React SPA) to access datacore, which means an HTTP API is needed.

Additionally, serialization is inconsistent: `base_data` uses JSON while `_custom_fields` uses TOON. The `_custom_fields` column name incorrectly uses the `_` prefix reserved for system metadata.

Current public API:
- `Store`: `put_entity()`, `get_active_entity()`, `get_active_model()`, `put_model()`, etc.
- `QueryEngine`: DuckDB+Arrow SQL over LanceDB tables
- Pre-production — no external consumers, no migration concerns

## Goals / Non-Goals

**Goals:**
- REST API layer (FastAPI) wrapping Store and QueryEngine — storage operations only
- Unify serialization: TOON for both `base_data` and `custom_fields`
- Rename `_custom_fields` to `custom_fields`
- CORS for admindash (`localhost:5174`)

**Non-Goals:**
- Document extraction (papermite's responsibility)
- Business logic or validation beyond key-conflict checks (consuming projects handle this)
- Authentication/authorization (future concern)
- Cross-tenant queries

## Decisions

### Decision 1: FastAPI in `src/datacore/api/` as thin wrapper

Add FastAPI as a thin HTTP layer over the existing Store and QueryEngine. The library API stays unchanged — the REST API just exposes it over HTTP. Package layout: `src/datacore/api/__init__.py` (app factory), `src/datacore/api/routes.py` (endpoint handlers).

**Endpoints:**

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/api/models/{tenant_id}/{entity_type}` | Fetch active model definition (wraps `Store.get_active_model()`) |
| POST | `/api/entities/{tenant_id}/{entity_type}` | Create entity with server-generated UUID (wraps `Store.put_entity()`) |

Entity ID is generated by the API layer (UUID) before calling `Store.put_entity()`. The frontend does not provide it.

Runs on its own port (e.g., `8081`).

### Decision 2: TOON for both columns

Switch `base_data` serialization from `json.dumps()` to `toon.encode()` in `Store.put_entity()`. Update all read methods (`get_active_entity()`, `get_entity_history()`) to use `toon.decode()` for `base_data`.

Update `QueryEngine._flatten_json_column()` to decode TOON instead of JSON. Rename to `_flatten_toon_column()` for clarity. The existing `_flatten_custom_fields()` method already decodes TOON for `custom_fields` — the pattern is proven.

Since datacore is pre-production, existing test data can be wiped. No migration script needed.

### Decision 3: Column rename `_custom_fields` → `custom_fields`

Update `ENTITIES_SCHEMA`, all Store methods, and QueryEngine references. Breaking change but pre-production with no external consumers.

### Decision 4: CORS configuration

Allow `localhost:5174` (admindash) and `localhost:5173` (papermite frontend). Use FastAPI's `CORSMiddleware`.

## Risks / Trade-offs

- **Breaking change to column name and serialization** → Zero risk, pre-production. Wipe test data.
- **FastAPI dependency added** → Minimal footprint. Only needed when running the API server, not when using datacore as a library.
- **No auth on API** → Acceptable for local development. Auth is a future concern for all NeoApex services.
