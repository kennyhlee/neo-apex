## ADDED Requirements

### Requirement: Model definition supports selection cardinality
The `FieldDefinition` for selection-type fields SHALL include a `multiple` boolean property that explicitly declares whether the field allows single selection or multiple selection. When `multiple` is `false` or absent, the field is single-select. When `multiple` is `true`, the field is multi-select.

#### Scenario: Field definition with explicit single-select
- **WHEN** a selection field definition has `multiple: false`
- **THEN** the field SHALL be treated as single-select across all frontends and the mapper

#### Scenario: Field definition with explicit multi-select
- **WHEN** a selection field definition has `multiple: true`
- **THEN** the field SHALL be treated as multi-select across all frontends and the mapper

#### Scenario: Field definition with missing multiple property
- **WHEN** a selection field definition does not include a `multiple` property
- **THEN** the field SHALL default to single-select (`multiple: false`)

### Requirement: Single-select fields render as radio buttons
Single-select selection fields SHALL render as a group of radio buttons in all form UIs (AdminDash DynamicForm, Launchpad FieldInput). Each option SHALL be a labeled radio button. Exactly one option may be selected at a time.

#### Scenario: Rendering a single-select field in AdminDash
- **WHEN** a selection field with `multiple: false` is rendered in DynamicForm
- **THEN** the field SHALL display as radio buttons with one button per option
- **AND** selecting one option SHALL deselect any previously selected option

#### Scenario: Rendering a single-select field in Launchpad
- **WHEN** a selection field with `multiple: false` is rendered in FieldInput
- **THEN** the field SHALL display as radio buttons with one button per option
- **AND** selecting one option SHALL deselect any previously selected option

#### Scenario: No selection state
- **WHEN** a single-select field has no value selected
- **THEN** no radio button SHALL be checked
- **AND** the field SHALL show a placeholder or empty state consistent with other empty fields

### Requirement: Multi-select fields render as checkboxes
Multi-select selection fields SHALL render as a group of checkboxes in all form UIs. Each option SHALL be a labeled checkbox. Zero or more options may be selected.

#### Scenario: Rendering a multi-select field in AdminDash
- **WHEN** a selection field with `multiple: true` is rendered in DynamicForm
- **THEN** the field SHALL display as checkboxes with one checkbox per option
- **AND** multiple options may be checked simultaneously

#### Scenario: Rendering a multi-select field in Launchpad
- **WHEN** a selection field with `multiple: true` is rendered in FieldInput
- **THEN** the field SHALL display as checkboxes with one checkbox per option
- **AND** multiple options may be checked simultaneously

### Requirement: Student model default cardinality
The Student entity model SHALL define the following selection field cardinalities as defaults:
- `grade_level`: single-select (`multiple: false`)
- `gender`: single-select (`multiple: false`)
- `status`: single-select (`multiple: false`)

#### Scenario: Student grade_level field definition
- **WHEN** the Student model definition is generated by the mapper
- **THEN** the `grade_level` field SHALL have `multiple: false`
- **AND** SHALL render as radio buttons in all form UIs

#### Scenario: Student gender field definition
- **WHEN** the Student model definition is generated by the mapper
- **THEN** the `gender` field SHALL have `multiple: false`
- **AND** SHALL render as radio buttons in all form UIs

#### Scenario: Student status field definition
- **WHEN** the Student model definition is generated by the mapper
- **THEN** the `status` field SHALL have `multiple: false`
- **AND** SHALL render as radio buttons in all form UIs

### Requirement: Program model default cardinality
The Program entity model SHALL define the following selection field cardinalities:
- `status`: single-select (`multiple: false`)
- `grade_levels`: multi-select (`multiple: true`)
- `days_of_week`: multi-select (`multiple: true`)

#### Scenario: Program status field definition
- **WHEN** the Program model definition is generated by the mapper
- **THEN** the `status` field SHALL have `multiple: false`

#### Scenario: Program grade_levels field definition
- **WHEN** the Program model definition is generated by the mapper
- **THEN** the `grade_levels` field SHALL have `multiple: true`
- **AND** SHALL render as checkboxes in all form UIs

#### Scenario: Program days_of_week field definition
- **WHEN** the Program model definition is generated by the mapper
- **THEN** the `days_of_week` field SHALL have `multiple: true`
- **AND** SHALL render as checkboxes in all form UIs

### Requirement: Consolidator preserves field cardinality
When the mapper consolidates duplicate entities of the same type, merging selection field options SHALL preserve the original `multiple` value from the field mapping. The consolidator SHALL NOT override cardinality to `true` when merging options.

#### Scenario: Merging single-select fields from duplicate entities
- **WHEN** two Student entities are extracted with `gender` options and the field is single-select
- **THEN** the consolidated field SHALL merge the options
- **AND** SHALL keep `multiple: false`

#### Scenario: Merging multi-select fields from duplicate entities
- **WHEN** two Program entities are extracted with `days_of_week` options and the field is multi-select
- **THEN** the consolidated field SHALL merge the options
- **AND** SHALL keep `multiple: true`

### Requirement: Model editor supports cardinality toggle
The Papermite model definition editor SHALL allow users to toggle between single-select and multi-select for any selection-type field via a checkbox or toggle labeled "Allow multiple selections".

#### Scenario: Toggling a field to multi-select in model editor
- **WHEN** a user checks the "Allow multiple selections" toggle for a selection field
- **THEN** the field definition SHALL set `multiple: true`
- **AND** the toggle state SHALL be reflected in the saved model definition

#### Scenario: Toggling a field to single-select in model editor
- **WHEN** a user unchecks the "Allow multiple selections" toggle for a selection field
- **THEN** the field definition SHALL set `multiple: false`

#### Scenario: Toggle only visible for selection type
- **WHEN** a field's type is not `selection`
- **THEN** the "Allow multiple selections" toggle SHALL NOT be displayed

### Requirement: Legacy data graceful handling
The UI SHALL handle data format mismatches between single-select and multi-select fields without errors.

#### Scenario: Single-select field with array value
- **WHEN** a single-select field encounters a stored value that is a JSON array
- **THEN** the UI SHALL display and use the first element of the array as the selected value

#### Scenario: Multi-select field with string value
- **WHEN** a multi-select field encounters a stored value that is a plain string (not a JSON array)
- **THEN** the UI SHALL treat the string as a single-element selection

### Requirement: Single-select stores plain string value
When a user selects a value in a single-select field, the form SHALL store the value as a plain string (not wrapped in an array).

#### Scenario: Saving a single-select value
- **WHEN** a user selects "Male" in a single-select gender field
- **THEN** the stored value SHALL be `"Male"` (string), not `["Male"]` (array)

### Requirement: Multi-select stores JSON array value
When a user selects values in a multi-select field, the form SHALL store the value as a JSON array of strings.

#### Scenario: Saving a multi-select value
- **WHEN** a user selects "Monday" and "Wednesday" in a multi-select days field
- **THEN** the stored value SHALL be `["Monday", "Wednesday"]` (array)

#### Scenario: Saving empty multi-select
- **WHEN** a user deselects all options in a multi-select field
- **THEN** the stored value SHALL be `[]` (empty array)
